# finish-pr

## 絶対禁止事項（最優先・本コマンド実行中は常時遵守）

以下は本コマンド実行中、常に遵守する。hooks等の外部機構に依存せず、本コマンド自身が確認する。

- **masterブランチへの直接コミット禁止**
- **force-push禁止**（`git push --force`および`--force-with-lease`は使用しない）
- **rebase禁止**（`git rebase`は使用しない。コンフリクトは`git merge`で解決）
- **コミットメッセージにCo-Authored-By行を含めない**
- **PR説明文にアシスタントツールへの言及を含めない**
- **masterへの直接マージ禁止**（必ずPull Request経由）
- **ブランチ名プレフィックス必須**（`feat/`, `fix/`, `chore/`, `refactor/`, `update/`, `posts/`のいずれか）
- **既存ブランチへの無関係な変更禁止**

上記のいずれかに該当する操作を検知した場合、即座に中断しユーザーに報告すること。

注意: hooksは主要パターンの抑止であり、`git reset --hard HEAD~N` + `cherry-pick` のような迂回ルートは拾えない。本コマンドのチェックリストとhooksの二重防御で守る。

## フロー

### ステップ1: ブランチ状態の確認

以下を実行して状態確認する。

- `git branch --show-current`
- `git status`

以下を確認:
- 現在ブランチがmasterではない
- ブランチ名が指定プレフィックスで始まる
- 未コミットの変更が残っていない

### ステップ2: コミット履歴の検証

`git log master..HEAD --format="%H %s%n%b"` を実行。

以下を確認:
- Co-Authored-By行が含まれていない
- コミットメッセージがプロジェクト規約に沿う
- 作業範囲外のファイル変更が含まれていない

### ステップ3: masterとの差分確認

`git diff master...HEAD --stat` を実行し、変更ファイル一覧がPRの目的と一致するか検証。

### ステップ4: テスト・リント実行

プロジェクトのテストコマンドとリントコマンドを実行し、全てパスすることを確認。

### ステップ5: PR作成

`.github/pull_request_template.md`の内容を読み込み、それに沿った説明を作成。

PRタイトルはブランチ名と同様のプレフィックスを付ける（例: `feat: ユーザー認証機能を追加`）。

PR本文にアシスタントツールへの言及を含めない。

`gh pr create --title "<プレフィックス>: <内容>" --body "<本文>"` で作成する。

### ステップ6: 作成後の確認

`gh pr view --web` でPRが正しく作成されたこと、レビュアー割当・ラベル付与が適切であることを確認。

## エラー時の対応

- 禁止事項違反を検知→即中断、ユーザーに違反内容と修正案を報告
- テスト失敗→修正してから再実行
- コンフリクト発生→masterを作業ブランチにmergeして解決（rebaseは使わない）

## 終了条件

- PRが作成され、URLがユーザーに共有された時点
- CIの結果を追いかける必要がある場合は、URLを共有してユーザー判断に委ねる
