# tdd

t-wada的なTest-Driven Developmentのサイクル（Red-Green-Refactor）に従って機能実装を行う。アサーションはVitestの`expect`に完全に寄せ、`node:assert`は使用しない。

## 起動時の絶対遵守事項

- 実装コードより先にテストを書く
- テストが失敗することを確認してから実装に進む
- 一度に扱うテストケースは1つに絞る
- masterへの直接コミット禁止、ブランチを切ってから作業すること
- テストランナー・アサーションは**Vitestに完全に寄せる**（`describe` / `it` / `expect` / `vi` は `vitest` から import）
- **`node:assert` の使用禁止**: `node:assert` / `assert` / `node:assert/strict` からの import は禁止。アサーションは必ず `expect` を使う
- テストリストを先に作り、優先順位をつけてから着手する（t-wadaの「TODOリスト」方式）

## サイクル

### Red: 失敗するテストを書く

1. 実装したい振る舞いを1つ決める
2. その振る舞いを検証するテストを書く
3. テストを実行し、失敗することを確認する
4. 失敗理由が「実装が存在しない」または「期待値と異なる」であることを確認

テストケース名は具体的な振る舞いを記述すること:
- 良い例: 「空配列のとき0を返す」「負の値を渡すとエラーになる」
- 悪い例: 「正しく動作する」「適切に処理される」

失敗するテストはVitestで書く:

```typescript
import { describe, expect, it } from "vitest";
import { sum } from "../sum";

describe("sum", () => {
  it("空配列のとき0を返す", () => {
    expect(sum([])).toBe(0);
  });
});
```

### Green: テストを通す最小実装

1. テストをパスさせる最小限のコードを書く
2. ハードコードでも構わない（後でリファクタリング）
3. `pnpm test:run` を実行し、全てパスすることを確認
4. 他のテストが壊れていないことも確認

### Refactor: コードを整える

1. テストが通った状態を維持しながらコード改善
2. 重複排除、命名改善、責務分離など
3. 各リファクタリング後にテスト実行
4. テストが壊れたら直前の状態に戻す

## サイクルごとのコミット

- Red-Green-Refactorの各段階で意味のある単位ごとにコミットすることを推奨
- ただしRedだけの状態で共有ブランチにpushしない

## 終了条件

- 要件に対応する全てのテストが書かれ、全てパスしている
- リファクタリング完了
- レビュー準備完了

## アンチパターン

- テストを後付けする
- 実装に合わせてテストを書く（テストが仕様駆動にならない）
- 複数の振る舞いを1つのテストにまとめる
- リファクタリングを省略する
- `node:assert` や他のアサーションライブラリを混在させる（Vitest の `expect` に統一すること）
- `describe` / `it` を `vitest` 以外から import する
- 実装詳細（内部プロパティ名や private メソッド呼び出し回数など）をテストする
