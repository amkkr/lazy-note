# Markdown 整形ルールの選定方針

> **最終更新**: 2026-05-16 / Issue #561 時点
> **想定読者**: 本リポジトリの `docs/**/*.md` を編集するコントリビュータ、
> および将来 Markdown 整形ツール導入を再評価する開発者

`docs/MILESTONES.md` / `docs/ANCHOR.md` 等の表セル幅整列は、Biome が
`.md` を整形対象としないため**手動で揃える運用**になっている。本ドキュメントは、
その「手動運用を継続する」判断と、その判断根拠・将来の再評価条件・運用ガイド
を集約する。

## 結論 (2026-05-16 時点)

**Markdown 整形ツール (markdownlint-cli2 / prettier) は導入しない。**
現状の「手動で表セル幅を揃える」運用を継続する。`docs/**/*.md` の数が
有意に増えたタイミングで再評価する (詳細は「再評価条件」節)。

### 採用しなかった案

| 案 | 内容 | 採用しなかった理由 |
| --- | --- | --- |
| 案 A | prettier 導入 (`pnpm exec prettier --write docs/**/*.md`) | Biome (formatter) と prettier の責務分割が混乱しやすい。`.md` のみ prettier、`.ts` / `.tsx` は Biome という分離は運用負荷が高い |
| 案 B | markdownlint-cli2 導入 | 表セル幅整列を強制するルールがそもそも存在しない (`MD058: tables should be surrounded by blank lines` 等はあるが、列幅整列は対象外)。AC「表整形手動 drift を防ぐ」を満たせない |
| 案 C (採用) | 現状維持 + 文書化 | 外部ライブラリ追加なし。対象ファイル数 4 件・表行 約 38 行と小規模で、レビュー時の目視で十分捕捉できる |

## 採用 (案 C) の判断根拠

1. **対象ドキュメントが小規模**: 2026-05-16 時点で `docs/` 配下の `.md` は
   合計 4 ファイル / 約 684 行、表行は約 38 行。レビューでの目視確認で
   drift を十分捕捉できる規模
2. **外部ライブラリ追加の方針**: 本リポジトリは「外部ライブラリ追加は原則
   しない / inline 実装で代替できないか先に検討」の方針を採っている。
   表整形のためだけに devDependency を増やす ROI は低い
3. **Biome との責務分割が混乱しない**: 案 A の prettier 導入は `.md` の
   みを対象とするとしても、開発者が `pnpm fmt` (Biome) と `pnpm fmt:md`
   (prettier) を使い分ける必要が出てしまう。`.ts` / `.tsx` の整形ルール
   と `.md` の整形ルールが別ツール由来になる構造は混乱の元
4. **Issue #561 自身が「優先度低、手動運用で当面回せる」と認識**: Issue
   本文の「優先度: 低」「手動運用で当面回せる」という認識と整合する判断

## 再評価条件

以下のいずれかに該当した場合、本判断を見直すこと:

- `docs/**/*.md` のファイル数が 10 件を超える、または合計行数が 2000
  行を超える (= 目視レビューでの drift 捕捉が困難になるライン)
- 表行 (`|` で始まる行) が累計 200 行を超える
- 手動 drift によるレビュー差し戻しが半年内に 3 回以上発生する
- Markdown 整形ツール側の事情が変わる (例: Biome が `.md` 整形を
  正式サポートする、prettier が Biome plugin として薄く統合される 等)

再評価する場合は新 Issue を切り、本ドキュメントを更新すること。

## 手動運用ガイド

### 表セル幅を揃える基本原則

1. **列ヘッダの直下の区切り行 (`| --- | --- |`) のハイフン数を、その列の最長
   セル幅に合わせる**
2. **各セルの右側はスペースで埋めて `|` の位置を揃える**
3. **長い説明文は表セルから本文の箇条書きに分解する**
   (例: `docs/MILESTONES.md` の `tone` 表は、補助マークの詳細を表外の
   「補助マークの注意」節に逃がしている)

### drift 検知のセルフチェック

`docs/**/*.md` を編集して PR を出す前に、以下を目視確認すること:

- 表内の `|` の縦位置が揃っているか
- 長い文がセル幅を肥大化させていないか (している場合は本文に逃がす)
- 区切り行 (`| --- | --- |`) のハイフン数が各列のセル幅と整合しているか

### エディタ補助

VS Code を使う場合、以下のいずれかの拡張で半自動整形できる
(導入は各開発者の判断、リポジトリへの強制はしない):

- [Markdown All in One](https://marketplace.visualstudio.com/items?itemName=yzhang.markdown-all-in-one):
  保存時に表セル幅を自動整列する機能あり
- [Markdown Table Formatter](https://marketplace.visualstudio.com/items?itemName=fcrespo82.markdown-table-formatter):
  選択範囲の表セル整列に特化
