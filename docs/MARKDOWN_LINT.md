# Markdown 整形ルールの選定方針

> **最終更新**: 2026-05-16 / Issue #561 時点
> **想定読者**: 本リポジトリの `docs/**/*.md` を編集するコントリビュータ、
> および将来 Markdown 整形ツール導入を再評価する開発者

`docs/MILESTONES.md` / `docs/ANCHOR.md` 等の表セル幅整列は、Biome が
`.md` を整形対象としないため**手動で揃える運用**になっている。本ドキュメントは、
その「手動運用を継続する」判断と、その判断根拠・将来の再評価条件・運用ガイド
を集約する。

## 結論 (2026-05-16 時点)

**Markdown 整形ツール (markdownlint-cli2 / prettier) は導入しない。**
現状の「手動で表セル幅を揃える」運用を継続する。`docs/**/*.md` の数が
有意に増えたタイミングで再評価する (詳細は「再評価条件」節)。

### 採用しなかった案

#### 案 A: prettier 導入 (`pnpm exec prettier --write docs/**/*.md`)

- 採用しなかった理由: 本リポジトリの「外部ライブラリ追加は原則しない / inline
  実装で代替できないか先に検討」という方針と整合しない。`.md` の表整形のため
  だけに devDependency を増やす ROI が、現状の小規模 (後述) では見合わない

#### 案 B: markdownlint-cli2 導入

- 採用しなかった理由: 表セル幅整列を強制するルールがそもそも存在しない
  (`MD058: tables should be surrounded by blank lines` 等はあるが、列幅整列は
  対象外)。AC「表整形手動 drift を防ぐ」を満たせない

#### 案 C: 現状維持 + 文書化 (採用)

- 採用理由: 外部ライブラリ追加なし。後述「判断根拠」のとおり対象ドキュメント
  が小規模で、レビュー時の目視で drift を十分捕捉できる

## 採用 (案 C) の判断根拠

1. **対象ドキュメントが小規模 (rfc 配下を含めた実測値)**: 2026-05-16 時点で
   `docs/` 配下 (rfc 含む) の `.md` は **5 ファイル / 約 800 行 / 表行 38 行**
   (うち rfc 配下 28 行)。レビューでの目視確認で drift を十分捕捉できる規模
   (内訳は本ドキュメント自身の行数を除く):
   - `docs/ANCHOR.md`: 244 行 / 表行 5
   - `docs/MILESTONES.md`: 132 行 / 表行 5
   - `docs/MARKDOWN_LINT.md`: 約 110 行 / 表行 0 (本ドキュメント。表は
     「採用しなかった案」を箇条書きに分解する方針のため意図的に 0)
   - `docs/rfc/editorial-citrus/README.md`: 40 行 / 表行 0
   - `docs/rfc/editorial-citrus/10-renewal-phase1-plan.md`: 277 行 / 表行 28
2. **外部ライブラリ追加の方針**: 本リポジトリは「外部ライブラリ追加は原則
   しない / inline 実装で代替できないか先に検討」の方針を採っている。
   表整形のためだけに devDependency を増やす ROI は低い
3. **Issue #561 自身が「優先度低、手動運用で当面回せる」と認識**: Issue
   本文の「優先度: 低」「手動運用で当面回せる」という認識と整合する判断
4. **rfc 配下は本ガイドの「手動運用ガイド」対象外として扱う**: rfc 配下
   (`docs/rfc/**`) は設計議論・Phase 計画の記録ドキュメントであり、本文の
   情報密度が高く、表セル幅 drift を厳格管理する優先度は低い。手動運用ガイド
   の対象は `docs/` 直下の運用ドキュメント (ANCHOR / MILESTONES 等) に絞る。
   ただし再評価条件 (次節) の **表行カウントには rfc 配下も含めて算出する**

## 再評価条件

以下のいずれかに該当した場合、本判断を見直すこと。
**数値は初期目安であり、実態 (レビュー差し戻し頻度・新規ドキュメント追加ペース)
を見て調整する**:

- `docs/**/*.md` (rfc 含む) のファイル数が 10 件を超える、または合計行数が
  2000 行を超える (= 目視レビューでの drift 捕捉が困難になるライン)
- 表行 (`|` で始まる行、rfc 配下を含む) が累計 200 行を超える
- 手動 drift によるレビュー差し戻しが半年内に 3 回以上発生する
- Markdown 整形ツール側の事情が変わる (例: Biome が `.md` 整形を
  正式サポートする、prettier が Biome plugin として薄く統合される 等)

### リスク: rfc 配下の表行が将来増える可能性

`docs/rfc/editorial-citrus/10-renewal-phase1-plan.md` の連番 (`10-`) からして、
今後 Phase 2 以降の追加で `11-...md` / `12-...md` 等が増える可能性がある。
2026-05-16 時点で **rfc 1 ファイルだけで表行 28 行 = 再評価条件「表行 200 行」
の 14%** を占めているため、Phase ドキュメントの追加ペース次第では再評価
条件に短期で到達しうる。Phase 追加 PR で表行が大きく増えた場合、本判断の
妥当性を都度確認すること。

再評価する場合は新 Issue を切り、本ドキュメントを更新すること。

## 手動運用ガイド

> **対象**: `docs/` 直下の運用ドキュメント (`ANCHOR.md` / `MILESTONES.md` 等)。
> `docs/rfc/**` は設計議論の記録のため、本ガイドの厳格運用対象外とする
> (ただし可読性を著しく損なう drift は PR レビューで指摘する)

### 表セル幅を揃える基本原則

1. **列ヘッダの直下の区切り行 (`| --- | --- |`) のハイフン数を、その列の最長
   セル幅に合わせる**
2. **各セルの右側はスペースで埋めて `|` の位置を揃える**
3. **長い説明文は表セルから本文の箇条書きに分解する**
   (例: `docs/MILESTONES.md` の `tone` 表は、補助マークの詳細を表外の
   「補助マークの注意」節に逃がしている)

### drift 検知のセルフチェック

`docs/**/*.md` を編集して PR を出す前に、以下を目視確認すること:

- 表内の `|` の縦位置が揃っているか
- 長い文がセル幅を肥大化させていないか (している場合は本文に逃がす)
- 区切り行 (`| --- | --- |`) のハイフン数が各列のセル幅と整合しているか
